## 🚨 Critical Risk 1: 法的コンプライアンスの実証可能性

### 1.1 契約書PDFの法的効力の検証

- [ ] **必須アクション**: 生成されるPDFテンプレートを**専任の弁護士**（労働法・下請法専門）にリーガルチェックさせる
  - 依頼先: [ ] 未定 / [ ] 確定済み（氏名: _______________）
  - 費用見積: _______________ 万円
  - 完了期限: _______________

- [ ] **必須項目の網羅性チェック**
  - 下請法で定められた項目（給付内容、報酬額、支払期日）が全て自動マッピングされているか
  - フリーランス保護法で追加された義務事項（業務指示の明確化、報酬確定時期など）に対応しているか
  - **実証方法**: 弁護士監修のチェックリストと照合

- [ ] **監査証跡（Audit Log）の改ざん不可能性**
  - Firestoreのセキュリティルールで「作成後の更新を禁止」する設計になっているか
  - 監査ログは「誰が・いつ・何を変更したか」を完全に記録できるか
  - **実証方法**: セキュリティルールの設計書を外部監査法人にレビューさせる

**⚠️ リスク**: この項目をスキップすると、システムが「法的に無効な契約書」を生成し続ける可能性があり、ユーザーから訴訟リスクが発生します。

## 🔐 Critical Risk 2: データ整合性とセキュリティ

### 2.1 マルチテナントデータ分離の実証

- [ ] **データリーク防止の設計確認**
  - `TenantID`による完全なデータ分離が実装されているか
  - クエリレベルでテナント間のデータ漏洩が不可能になっているか
  - **実証方法**: ペネトレーションテスト（外部セキュリティ企業に依頼）

- [ ] **個人情報保護法（GDPR/日本の改正個人情報保護法）への準拠**
  - 顧客の採寸データ（身体情報）の暗号化が実装されているか
  - データ削除リクエスト（忘れられる権利）に対応できる設計か
  - **実証方法**: コンプライアンスチェックリストへの記載

**⚠️ リスク**: データ漏洩が1件でも発生すると、事業継続が困難になります。

### 2.2 決済トランザクションのACID保証

- [ ] **PostgreSQLでの決済処理の実装方針**
  - 決済処理は必ずPostgreSQLで行い、Firestoreは補助的な用途のみとする
  - トランザクション処理で二重決済や不整合が発生しない設計か
  - **実証方法**: 負荷テストで同時決済100件を実行し、エラー0件を確認

**⚠️ リスク**: 決済の不整合は即座に財務損失と顧客信頼の失墜につながります。

### 3.1 MVPでの収益化仮説の検証

  - 初回課金タイミング: [ ] 登録時 / [ ] 10件目の発注時 / [ ] その他: _______________
  - **決定者**: CEO（井上寛太）による承認

- [ ] **ユニットエコノミクスの初期設定**
  - 月額料金プラン: [ ] 基本プラン ¥______ / [ ] プロプラン ¥______
  - トランザクション手数料: [ ] 0% / [ ] ____% （Phase 3のFintech機能用）
  - **実証方法**: LOI獲得済み10社へのヒアリングで「支払ってもらえる価格帯」を検証

**⚠️ リスク**: 収益化方針が不明確だと、開発コストだけが積み上がり、事業継続が困難になります。

### 3.2 顧客獲得コスト（CAC）とLTVの試算

- [ ] **CAC試算**: 1社あたりの獲得コスト = _______________ 万円
- [ ] **LTV試算**: 1社あたりの生涯価値 = _______________ 万円
- [ ] **LTV/CAC比率**: [ ] 3倍以上（健全） / [ ] 3倍未満（要改善）

**⚠️ リスク**: LTV < CAC の状態が続くと、事業は持続不可能です。

## 🎯 Critical Risk 4: ユーザー体験（UX）の実証可能性

### 4.1 「現場の指」への適合性

- [ ] **文字入力の最小化**
  - 発注入力画面で、キーボードを出す必要がある項目が3つ以下になっているか
  - QRコードスキャンで生地選択が可能か
  - **実証方法**: 実際のテーラー（LOI獲得済み企業）でのユーザビリティテスト

- [ ] **音声入力の精度**
  - 音声認識API（例: Google Cloud Speech-to-Text）の日本語認識精度が90%以上か
  - 認識エラー時の修正フローが直感的か
  - **実証方法**: 職人10名で実際の採寸データを音声入力し、エラー率を測定

**⚠️ リスク**: 入力が煩雑だと、アプリは使われず、紙の伝票に戻ってしまいます。

### 4.2 オフライン動作の実証

- [ ] **オフライン時の動作確認**
  - 工場の電波が弱い環境でも、発注データをローカルDB（Hive/Drift）に保存できるか
  - 電波復旧時に自動同期が機能するか
  - **実証方法**: 実際の工場（協力工場）で電波を遮断した状態での動作テスト

**⚠️ リスク**: オフライン動作が不完全だと、現場で使えないシステムになります。

## 📦 Critical Risk 5: 在庫データの信頼性

### 5.1 在庫タイムラグ問題への対策

- [ ] **Safety Stock（安全在庫）ロジックの実装**
  - CSVバッチ連携の場合、システム上の在庫が「3.2m以上」でも、実際は「△（要確認）」と表示する設定が可能か
  - ユーザーに「在庫あり（◎）」と表示する閾値を安全側に設定できるか
  - **実証方法**: 問屋との連携テストで、実際の在庫切れ発生率を測定

- [ ] **在庫データ連携の責任範囲の明確化**
  - 在庫データの不整合による機会損失の責任は誰が負うか（TailorCloud vs テーラー）
  - 利用規約に「在庫情報は参考値であり、確定在庫は問屋への直接確認が必要」と明記するか
  - **決定者**: CEO + 法務（弁護士）による承認

**⚠️ リスク**: 在庫切れトラブルが多発すると、テーラーの信用問題となり、解約の最大要因になります。

## 🏗️ Critical Risk 6: 技術的負債とスケーラビリティ

### 6.1 マイクロサービス化への準備

- [ ] **初期実装でのモノリスの範囲**
  - Phase 1では、すべてのAPIを1つのGoサービス（Cloud Run）で実装
  - Phase 3以降で、決済サービス・在庫連携サービスを分割する設計になっているか
  - **実証方法**: アーキテクチャ設計書のレビュー

- [ ] **データベースマイグレーション戦略**
  - PostgreSQLのスキーマ変更を安全に実施する手順が確立されているか
  - Firestoreのデータ構造変更時の移行手順が明確か
  - **実証方法**: 開発環境でのマイグレーションリハーサル

**⚠️ リスク**: 将来の拡張性を考慮しない設計は、技術的負債となり、後で全面的な作り直しが必要になります。
